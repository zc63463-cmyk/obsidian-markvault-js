## 7项已确认的架构决策（按优先级排序）

### 1. MindFlow 是"认知工作台"（非投影面） ⭐⭐⭐
- **确认内容**：导图不是对已有标注的被动排列，组织过程会**产生新认知**
- **场景举例**：解题步骤可视化、同一知识点跨题对比、多解法对比发现深层结构
- **数据价值**：导图作为历史训练数据，记录"人怎么思考"的认知过程，不仅"人知道了什么"

### 2. structureType 字段 — 认知结构类型 ⭐⭐⭐
- **确认内容**：导图需要 `structureType` 字段标记用户当时的认知意图
- **类型**：`flow`(流程图) | `skeleton`(脉络图) | `hierarchy`(组织结构) | `process`(过程图) | `fishbone`(鱼骨图) | `freeform`(自由混合)
- **意义**：不同 structureType 对应不同认知模式，AI 分析时知道"用户在用什么方式思考"
- **存储**：frontmatter `mindmap.structureType`，与 `layout` 独立（语义 vs 视觉）

### 3. MindFlow 三种使用模式 ⭐⭐⭐
- **模式① 纯导图**：独立存在，不依赖标注，普通思维导图
- **模式② @引用标注**：导图节点引用已有标注（annotationRef），点击可查看详情
- **模式③ 导图节点被标注**：text=节点内容，note=认知描述（"这题很有启发意义"属于note不是text）
- **关键**：模式③是在导图组织过程中即时捕获认知火花，非低频操作

### 4. byAnnotationRef 反向索引 ⭐⭐
- **确认内容**：需要从标注 UUID 反查它出现在哪些导图的哪些节点
- **用途**：复习时查看"这个标注在不同导图里被怎么组织过" → 多维归属可视化
- **实现位置**：MarkVault annotation-store.ts，与 byDocType/byNodeId 同层索引

### 5. Relation note 一等公民 ⭐⭐
- **数据层就绪**：`AnnotationRelation.note` 字段已存在
- **交互需强化**：创建 relation 时提示填写 note；反向链接查看时 note 是主要信息
- **导图-relation 关系**：不在导图里建立 relation（避免UI层误操作污染数据层），导图是发现现场 → 用户确认后 add relation

### 6. 多维连接体系 ⭐⭐
- 语义关系(27种，横向网状) + MindFlow结构(纵向树状，多结构类型) + 认知状态(flags)
- **结论**：核心资产不是"27种关系"，是"多维连接体系" — 三种连接捕捉三种不同的认知信息

### 7. 开发优先级 ⭐⭐
- **当前阶段**：PDF 和 MindFlow 并行开发
- **下一阶段**：考研备考期间用系统 dogfooding → 积累真实使用数据
- **远期**：认知层迭代（FSRS/迷雾模型/虚拟讲师）→ 平台化（3-5年后）

## 开发注意事项

1. **数据层统一，渲染层分离**：Annotation 对象在编辑器/导图/PDF 中共享，但渲染方式各自最优
2. **MarkVault 可选依赖**：MindFlow 在 MarkVault 未安装时正常降级运行（纯 MD-Seed + Free）
3. **docType 路由独立实现**：不依赖 PDF 扩展完成，MindFlow 可直接注册 `docType: 'mindmap'`
4. **relation 不在导图中建立**：导图是认知过程现场，关联发现后用户手动 add relation，避免UI误操作
5. **测试先行**：项目已有 541/541 全绿的 tsx 测试套件，新代码必须有对应测试
