## 修复 CodeMirror 双实例报错

### 问题
- 控制台报错：`Unrecognized extension value in extension set ([object Object]). This sometimes happens because multiple instances of @codemirror/state are loaded`
- 原因：之前手动用 `npx esbuild src/main.ts --bundle` 构建，没走 `esbuild.config.mjs` 的 external 配置，导致 `@codemirror/state` 和 `@codemirror/view` 被内联打包进 main.js
- Obsidian 运行时已加载一份 @codemirror/state，插件又自带一份 → `instanceof` 检查失败

### 修复
- 用 `node esbuild.config.mjs production` 正确构建（external 配置生效）
- 修复后 main.js 只含 `require("@codemirror/state")` 而非内联源码
- **教训**：Obsidian 插件构建必须走 esbuild.config.mjs，不能手动 esbuild

## 图谱自适应优化 v5.5（参考业界最佳实践重构）

### 参考来源
- force-graph 官方 API：nodeCanvasObject 的 globalScale 参数用于缩放自适应
- d3-force 文档：forceLink.distance() 支持函数（per-link 动态距离）
- d3-force 文档：forceManyBody.strength() 支持函数（per-node 斥力）
- d3-force-3d：force-graph 的内部依赖，包含 forceCollide（碰撞检测）
- obsidian-graph-view-text-size 插件：Canvas 字体大小调整思路

### 核心改进（v5.4 → v5.5）

| 改进 | v5.4 | v5.5 |
|------|------|------|
| **连接线距离** | 全局统一 `suggestedLinkDistance` | per-link `_distance`，基于两端标签长度独立计算 |
| **标签截断** | 数据层截断（MAX_LABEL_LEN=20） | 渲染层动态截断（根据 globalScale 三档策略） |
| **字体大小** | `min(16,max(8,13/globalScale))` 线性 | `min(14,max(7,12/sqrt(globalScale)))` 平滑曲线 |
| **标签可见性** | globalScale > 0.7 全显示 | 三档：>0.6 截断12字，>1.5 截断24字，<0.6 隐藏 |
| **节点斥力** | 全局统一 -120/-150/-200 | per-node `-80 - degree*15`（高度节点斥力更大） |
| **碰撞检测** | 无 | forceCollide，含标签区域的碰撞半径 |
| **标签背景** | 方形 fillRect | 圆角矩形 quadraticCurveTo |

### 文件变更
- `graph-data-builder.ts`: 移除 suggestedLinkDistance，新增 per-link `_distance`；label 保留原始值
- `RelationGraphView.ts`: drawNode 三档渲染 + applyAdaptiveForces per-link/per-node + forceCollide
- `graph-data-builder.test.ts`: label truncation 测试更新

## 控制台报错审查: RangeError: Field is not present in this state (08:02)
- **报错位置**: app.js → Obsidian 核心 `saveHistory()` → `beforeUnload()` → `onUnloadFile()`
- **根因**: Obsidian 核心问题，非 MarkVault 插件直接导致
- **机制**: 文件关闭时 CM6 尝试保存 undo 历史，但此时 state 可能已被部分销毁导致 Field 查找失败
- **MarkVault 关联**: 插件注册了 `regionCacheUpdatedEffect` (StateEffect) 和 2 个 ViewPlugin，会在 CM6 state 中创建对应 Field。若这些 Extension 在插件卸载/视图切换时未被正确清理，state 中仍引用已销毁的 Field，触发 RangeError
- **当前防护**: `requestRegionLayerRedraw()` 已有 `view.destroyed` 检查 + try-catch 兜底
- **建议**: 在 `onunload()` 中显式置空 `activeEditorView`，减少异步 dispatch 到已销毁 view 的概率
